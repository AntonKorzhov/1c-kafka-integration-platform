# Архитектурное решение

Интеграция реализована по событийной схеме:

```text
1С HTTP API
        │
        ▼
integration-service
        │
     Kafka
        │
        ▼
consumer-service
        │
        ▼
 PostgreSQL
```

Producer получает данные из HTTP API 1С, преобразует их в единый формат событий и публикует в Kafka. Для каждого типа справочника используется отдельный Kafka topic:

- `1c.ownership_forms.v1`;
- `1c.counterparties.v1`.

Producer сначала публикует формы собственности, а затем контрагентов. Это уменьшает вероятность появления контрагента раньше связанной записи формы собственности. Если зависимая запись ещё отсутствует, consumer повторяет обработку после временной ошибки внешнего ключа.

Сообщения Kafka имеют стабильный ключ — UUID объекта 1С (`payload.id`). Благодаря этому повторная публикация одного и того же объекта не приводит к появлению дубликатов.

Consumer сохраняет данные в PostgreSQL с использованием механизма `INSERT ... ON CONFLICT DO UPDATE`, поэтому повторная обработка сообщений является идемпотентной. Если в Kafka повторно поступает более старое событие, оно не перезаписывает более новое состояние записи. Удаление реализовано как мягкое удаление посредством установки флага `deleted`.

Для инкрементальной синхронизации producer хранит watermark каждого справочника в таблице `sync_state`. После успешной публикации сообщений в Kafka watermark обновляется. При следующем запуске producer повторно считывает одну предыдущую секунду диапазона, что позволяет избежать потери изменений на границе временных меток.

Для обработки ошибок реализованы:

- ограниченное количество повторных попыток обработки;
- Dead Letter Queue (`1c.integration.dlq.v1`) для необрабатываемых сообщений;
- структурированные JSON-логи сервисов;
- централизованный просмотр логов через Grafana, Loki и Alloy.
